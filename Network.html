Introduction

I had just a Desktop PC running Ubuntu linux and nothing else. Suddenly I seem to have acquired a laptop (dual boot windows 8 and Ubuntu) and an android tablet on which I installed androidVNC. The tablet (a Motorola Xoom MZ604 with 32GB internal SD card and 1GB RAM) that is set up so it can run openSuSE linux. Oh yes, and a wireless enabled printer as well. The obvious thing to do is network them!

This article tells you how I got on...

Finding out a bit more

In this section, we shall start to look at some basic linux/UNIX network commands. The first is ifconfig (interface configuration). This is a part of UNIX. Though it's used to configure networks, just doing "ifconfig" in a terminal will give you some network information. By "a terminal" I really mean "in a terminal emulator window" if you want me to a bit precise. In a UNIX terminal, you type a command (such as env or pwd or ls) and hit the enter button. It's useful to get used to using the "command line" in a terminal. The iwconfig command is similar to ifconfig but specialised for wireless networks: it isn't part of linux/UNIX in itself. The iwconfig command xomes with the "wireless-tools package". Any linux distribution will allmost certainly have installed the wireless-tools package when you installed linux. So you almost certainly have iwconfig. If not, install that wireless-tools package now.

You may or may not have the "lynx" or "curl" programs installed. Why not imake sure you have both! Curl is a command line tool that can transfer data to and from a server, and lynx is a text based web browser you can run in a terminal. We can get our IP address as seen from the outside world using either. They might sound quaint because the access the web from a terminal, but both are quite powerful tools. Doing either "curl http://ipecho.net/plain; echo" or "lynx --dump http://ipecho.net/plain" as terminal commands will give you your external IP address. Obviously the ipecho.net site does exactly that: it echos the IP that sends it a request. That is to say, this is the router IP address as far as the outside world is concerned. If I am on BigBeast, I can see my IP address as far as the local network is concerned doing "ping BigBeast" in a terminal. Just < Ctrl > to stop it.

Setting Up Telnet, Ssh, and Ftp

There are two easy ways to remote login from one machine to another via a terminal. These are telnet and ssh. There's an easy to use terminal based file transfer program called ftp as well. Suppose you are on RoverBeast, and telnet is up and running. You type in "telnet BigBeast" at a terminal, and there you are. You log in to BigBeast, you now have a terminal on RoverBeast's windows acting as a terminal on BigBeast. You can run a program on BigBeast for instance, or move and edit files on BigBieast's hard drive. Here BigBeast is the "host" and "RoverBeast" is the client. Using telnet, it should be easy to get the host machine to display an X-Windows program output (being run on the host) on the client machine. For some reason I have problems doing this on my local network: never mind, we can use ssh (secure shell) and that uses dual key encryption to boot. Nevertheless, I shall mention what should have worked anyway.

So, telnet is a nice easy terminal based way to get into another machine in your local network. You can telnet to machines not on your local network, like another company for instance. Telnet is standard UNIX/linux, so you don't need to install anything to be a telnet client. Let's try it and see what happens! Suppose I am on BigBeast and I am "in a terminal". If I do "telnet Roverbeast" from a terminal, I get

Trying 192.168.1.68...
telnet: Unable to connect to remote host: Connection refused

So, the good news is that telnet understands that RoverBeast is that 192.168.1.68 IP address. It has got a message back that the connection is refused. I get the same if I try and telnet into BigBeast from RoverBeast. It understands the IP address is 192.168.1.64 from the name BigBeast, but the connection is refused. What's the problem? Both machines are automatically telnet clients but not telnet servers. On Ubuntu all we have to do is open up system settings, and then install telnetd. It will be just as easy on any linux distribution. When you reboot, it will be running the telnet "daemon" telnetd. If you had checked out the file /etc/inetd.conf you will have seen that telnetd was in that initialisation file even before you installed telnetd. Here is my /etc/inetd.conf file line for telnet.

#:STANDARD: These are standard services.
telnet stream tcp nowait telnetd /usr/sbin/tcpd /usr/sbin/in.telnetd

So, now if I do "telnet RoverBeast" from a terminal on BigBeast, I get to log into RoverBeast with username and password. I can use a command line editor, run programs, and pretty well anything I want. To log out of RoverBeast, all I do us use "exit", and the terminal is handed back to BigBeast. But (for me somehow) there is a snag...

So, suppose I have a program that opens up a window. For instance I have a simple program that allows you to view, rotate, and move a dodecahedron in a window. It's called Dodec. If I am on RoverBeast, and run Dodec, all is fine. If I do "env | grep DISPLAY" in a terminal, it tells me that the environment variable DISPLAY is set to ":0" (it could be :0.0 on other machines). Now, suppose I want to run Dodec on BigBeast from a terminal on RoverBeast. I can telnet into BigBeast and try and run Dodec, but I shall get error messages, or nothing will happen. To see why, I now do "env | grep DISPLAY". This might not even output anything at all, in which case the DISPLAY variable hasn't even been set yet. The terminal is running the "bash shell", in which case I can set the DISPLAY variable by doing "export DISPLAY=:0". (On a c-shell terminal the same thing is done using "setenv DISPLAY=" instead of "export".) Now if I run Dodec from the terminal on RoverBeast, I see the window displaying a dodecahedron appear on BigBeast: not on the client machine! Now, we want the output of Dodec being run on BigBeast to appear on RoverBeast's screen. To do this we need to allow BigBeast to output to RoverBeast's X-Windows display. At a terminal on RoverBeast, this is done by "xhost + BigBeast" (you might have to install xhost). Now I tried (for "export PATH=) RoverBeast:0, 192.168.1.68:0, RoverBeast:0.0, and nothing worked! I have used this xhost + and DISPLAY thing regularly on other networks, so I can't figure out what is wrong here. Never mind, it might work for you. I can still use telnet to do anything apart from running programs displaying windows on the client. I mentioned ssh earlier. With ssh, we don't need anything like xhost, and we don't have to mess about with environment variables. Not only does it take care of all that stuff for you, but uses dual key encryption between the two machines. This means we have to set up the encryption keys. Let's see how that's done next.

First thing off, we need to install the ssh client and server on both machines (and reboot). Doing this gives you ssh-keygen and ssh-copy-id as well as ssh server and client. Now, I made a directory called ".ssh" on BigBeast's home directory. It must be the home directory (a.k.a ~), and it must be called .ssh. You might have a ~/.ssh directory already. Either way, I changed directory to .ssh in a terminal, and ran the command

ssh-keygen -t rsa -C "BigBeast"

This creates a private key ~/.ssh/BigBeast, and a public key called ~/.ssh/BigBeast.pub. Obviously, you will choose different names. I did the same thing (almost) on RoverBeast, so RoverBeast's ~/.ssh directory now has a private key "RoverBeast" and a public key "RoverBeast.pub. From BigBeast, I did

ssh-copy-id BigBeast.pub username@RoverBeast"

When you do your equivalent, username is whatever your username is, and be sure to type "yes" and not "y". This will copy BigBeast's public key to RoverBeast. I used the same command to copy RoverBeast's public key to BigBeast. If you do the equaivalent, ssh can now handle dual encryption messages between the two machines. So, on RoverBeast, if I do "ssh BigBeast" I can log in with my password, and everything looks just like telnet. Now, what If I want to run that Dodec program so it appears on RoverBeast's screen? To do this I have to log out (use "exit" again as you do in telnet). Now, we are going to use ssh again, this time with an option "-X". That is, I type "ssh -X BigBeast". This option will allow BigBeast to display windows on RoverBeast's screen. So, now I get that dodedechedron window up on the right computer.

The last thing we do in this section is to enable ftp (file transfer protocol) between our two machines. As you may have guessed already, we have to install an ftp daemon on both machines. On Ubuntu, Though the server daemon is called vsftpd, the command to use is still ftp. Now, if I do "ftp BigBeast" from RoverBeast, I can log in to BigBeast and I see an ftp prompt in the terminal. Do the same on your network, and ftp to the other (or another) machine on the network, and type "help" at the ftp prompt. You will see commands like put, mput, get, mget and so on. Do "help mget" and it tells you that mget can pull across multiple files. With my two machines, I can use commands like cd, pwd, ls, and so forth to navigate around BigBeast. Suppose you want to get a load of text files with a ".txt" extension. from another machine. Use ftp OtherMachine (say) and use the navigation commands to get to where the text files are. Then (at the ftp prompt) do "mget *.txt" to copy all the files with a ".txt" extension in that directory over onto RoverBeast.

So now we have terminal based communication between the two machines vi telnet, or ssh, or ssh -X, and file transfer to and fro using ftp. Well, it's a start...

Samba File Sharing and Printer Network

From the first of the "Useful links" below, "Samba (http://www.samba.org) is software that allows non-Windows computers to act like Windows computers on a network by implementing Windows file and printer sharing protocols". Samba enables us to share files across machines in a network. It's attractiveness is that it's "cross platform". Which means for instance, that we can share files between linux and MS windows. It is based on SMB (Server Message Block) and CIFS (Common Internet File System) whatever they happen to be.

So, I shall start on my desktop machine (BigBeast) running linux. I shall suppose I want to share some artwork stored in a directory called "backgrounds". If I right click on this directory in the file manager, I see "sharing options". If I tick "share this folder" it tells me I "Sharing Service Not Installed" and gives me the option to install samba. Let's go for it? Along the way, it asks if I want "libpam-smbpass" and I say yes. Then I say yes to "restart session" (this doesn't restart the computer). I have just installed samba. We now have (as terminal commands) eventlogadm, mksmbpasswd, nmbd, pdbedit, profiles, smbcontrol, smbd, smbstatus, smbstatus.samba3. Now I can right click on backgrounds, go to share options, and click on "create share". Over on my laptop (RoverBeast running linux), I have the directory "mybackgrounds". I do the same thing so samba and libpam-smbpass are installed there too. On both machines, I had the option of allowing others to create and delete files in the shared directory, and on both machines I noticed a warning that the share name was too long. For the moment, I have not allowed creating or deleting files. On both machines I install samba-config-samba (a GUI for the GNOME desktop).

So, the burning question of the day is, can I share files between the two linux machines? The first thing to do is launch that GUI. I go through to the add symbol for files and add backgrounds. (Yes, I thought it would be there already too.) Once it is there in the list of shares, you can highlit it, click on files again, and choose properties. Here, you can modify the description of the share, make the share writable (or not), and visible (or not). This was under "Basic". Under "Access" I can either allow certain users to have access, or I can allow access to everyone. I shall choose the latter for the momement, and make the share writable. This is on both machines. To add a password for a user I could use (as root) smbpasswd -a username, and then (as root) /etc/init.d/samba reload, but I can set user passwords using the GUI. Just for the tryout I use the rather stupid password "sambapass". The "help" in the GUI isn't bad!
